  1. Introduction

  1. Purpose[a] 

This Software Requirements Specification (SRS) specifies the requirements of a new web application in development called Relativiti, which will be used in Comp 330: Software Engineering. This document will be useful for the clients to ensure all specifications and requirements are conducive as mentioned by the software engineer to design the system.

  1. Document Conventions[b]

This document will use Arial font for all text. Bodies of text will have 11 font text. In some cases when outlining a series of ideas, a bulleted list will be used. Comments from the authors that are meant to pose questions or display information relevant to building the document will be displayed utilizing the comment function through the google docs app. Main section titles will have 18 font in bold, while subsection titles will have 14 font in bold.

  1. Intended Audience and Reading Suggestions[c]

  1. Clients: The users of the system will have a clear perspective as to the software and hardware specifications. Clients are encouraged to read Sections 1,2. If more technical information is desired by the client, then it is suggested that they read section 3, and for a full-depth explanation of the full system the rest of the sections.
  2. Developers: The developers of this project will be able to attain a rapid understanding of the methodology and the implementations of the various aspects of this system. It is strongly encouraged that developers read this whole document.
  3. Colleagues:Fellow students in Comp 330 will gain a better perspective of how we envisioned our project’s development and our team’s implementation of the initial idea. Colleagues are encouraged to read sections 1,2 and for more technical information, sections 3,4, and 5.

  1. Product Scope[d]

Scope of this project can include musicians, composers, or anyone with a use for sheet music or tuning of an instrument. The benefits that this application will provide is a quick return of the live audio into a workable and readable piece of sheet music. It will also double as a way for musicians to be able to tune their instruments or match notes correctly.

 Our goal with this project is to make it easier for people who don’t know how to write sheet music correctly, want to simplify the process, or don’t just have the time for it. This is also useful for musicians, dabblers, and artists who want to remember a specific piece while they are playing, but may not have the time to stop and write down the score. This makes for a more agile way of saving and distributing your work.

  1. Overall Description

  1. Product Perspective[g]

This web application is currently is a new system in development. Our sheet music functionality is currently in its infant stages. In regards to the tuning functionality- our application is a response but also an enhancement to the apps currently in the mobile market. We would like to present a product with expanding functionality in tuning; to provide more than just the basic guitar string notes which in large is the scope of tuning functionality available on the market for free apps.

  1. Product Functions[h]

  1. With this web application, users will be able to record sound input. This sound input will be obtained by usage of a microphone (or other external hardware).
  2. The user will be able to specify through the interface which functionalities they would like to use - tune a device, record a new piece of music, or view previously recorded music.
  3. Tuning a device
  1. Users can select which note they would like to tune to; notes that can be chosen are the widely used concert notes .[i]
  2. Users can play instruments or use their voice as input into the device. The device will parse this sound input in real time.
  3. The sound input will be displayed in real time which frequency it is displaying. It will give visual cues to how close it is to the chosen note.
  1. Recording Music and Conversion to Sheet Form
  1. Users can select when they would like their recording to begin and end
  2. After the recording is finished, the app will read the sound file, parse the information, and display it in sheet music form.
  3. For each composition the app will distinguish the time signature and key.
  4. For each note played the app will distinguish the note duration and its pitch.
  5. The sheet music will be displayed at the end of the recording as a pdf or midi.
  6. The user can modify certain features of the piece - the title of the music, comments (anything else?)
  7.  The user can save the piece to view later or delete it.
  8. The user can export or share piece of music through email or save it to the device.
  9. Can the user play back the recorded melody?
  1. Viewing Previously Recorded Music
  1. The User can access saved pieces, modify certain attributes; name, comments, (anything else? perhaps time signature)
  2. The user can share the PDF’s (or other visual file format) via email or download them to the device.
  3. The user can delete the sheet music from the library.
  4. can the user play back the recording?
  1. Other considerations like log-in?

  1. Performance Requirements[an]

This web application should perform as close to real time as possible. Undesirable performance would include having inconsistent and lengthy wait times for basic functionalities. For the scope of this project within its first release, we are looking to have this web application perform to these standards under a small load. More specifically a single user at one time. Multiple users is not currently a huge consideration.
